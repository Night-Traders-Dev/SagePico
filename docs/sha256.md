# SHA-256 Hardware Accelerator

RP2350 hardware SHA-256 hashing engine with PIO-accelerated data feeding.

## Usage

```
>>> sha256("hello")
2cf24dba5fb0a30e26e83b2ac5b9e29e1b161e5c1fa7425e73043362938b9824
```

### Pure Sage API

```sage
import sha256
let digest = sha256.hash("hello world")
```

### FFI (on-device)

```python
ffi_call(pico, "sha256", ["hello world"])
```

## Hardware Details

- **Engine**: RP2350 SHA-256 hardware accelerator (`sha256_hw` at `0x52000000`)
- **Block size**: 64 bytes (512 bits) per round
- **Digest size**: 32 bytes (256 bits) → 64 hex characters
- **Endianness**: Little-endian output (configured via `SHA256_LITTLE_ENDIAN`)
- **DMA request**: `DREQ_SHA256 = 54` — triggers DMA transfers when accelerator is ready for data

## Implementation: Three Data Paths

The bridge automatically selects the fastest available data path:

### 1. Word-at-a-Time (Default)

```c
// Feed 4 bytes at once via sha256_put_word()
while (i + 4 <= len) {
    uint32_t word = (data[i] << 24) | (data[i+1] << 16) |
                    (data[i+2] << 8) | data[i+3];
    sha256_put_word(word);
    i += 4;
}
```

**Throughput**: ~8-16 MB/s. CPU polls `sha256_wait_ready_blocking()` between words.

### 2. DMA Streaming (>256 bytes)

```c
sha256_set_dma_size(len);
dma_channel_configure(dma_ch, &c,
    &sha256_hw->wdata,   // DMA writes directly to SHA-256 data register
    data,                // Source memory
    (len + 3) / 4,       // Word count
    true);               // Start immediately
dma_channel_wait_for_finish_blocking(dma_ch);
```

**Throughput**: Bus-speed limited (~100+ MB/s). DMA handles 32-bit transfers triggered by `DREQ_SHA256` — the accelerator signals when it's ready, DMA pushes a word, repeat. CPU sleeps during the entire hash.

**Fallback**: If no free DMA channel, falls back to word-at-a-time.

### 3. PIO-Accelerated Streaming (Architecture Proposal)

The RP2350's PIO state machines can act as a dedicated hardware data pump between DMA and SHA-256, eliminating the CPU from the feeding loop entirely.

#### How It Works

```
┌─────────┐    DMA     ┌─────────────┐    32-bit    ┌──────────────┐
│  SRAM   │ ────────→  │  PIO TX FIFO │ ──────────→  │  SHA-256 HW  │
│ (data)  │  DREQ_PIO  │  (8 words)   │  SM loop    │  (wdata reg) │
└─────────┘            └─────────────┘             └──────────────┘
```

#### PIO Program (16-bit instructions, 4 words)

```asm
.program sha256_pump
.side_set 0

    pull block          ; Wait for 32-bit word from TX FIFO
    out pins, 32        ; Write 32-bit word to data pins (SHA-256 wdata)
    jmp sha256_pump     ; Loop forever
```

The PIO state machine:
1. Blocks on `pull` until DMA pushes a word into the TX FIFO
2. Outputs the 32-bit word in a single cycle via the `OUT` instruction
3. Loops immediately — ready for the next word

#### DMA Configuration

```c
// DMA channel feeds PIO TX FIFO from memory
dma_channel_configure(dma_ch, &c,
    &pio0_hw->txf[sm],   // Destination: PIO SM TX FIFO
    data,                 // Source: input buffer
    (len + 3) / 4,        // Word count
    false);               // Don't start yet

// Second DMA channel paces transfers from PIO to SHA-256
// (or use DREQ_SHA256 directly on the first channel)
```

#### Performance Comparison

| Method | CPU Load | Throughput | Use Case |
|--------|----------|------------|----------|
| `sha256_put_byte()` | 100% polling | ~1-2 MB/s | Tiny data (<64 bytes) |
| `sha256_put_word()` | 100% polling | ~8-16 MB/s | Small data (<256 bytes) |
| DMA → SHA-256 | 0% (sleeps) | ~100 MB/s | Bulk data (>256 bytes) |
| DMA → PIO → SHA-256 | 0% (sleeps) | ~100 MB/s | Bulk + flexible routing |

#### When PIO Helps

Direct DMA → SHA-256 is already optimal for simple bulk hashing. The PIO path adds value when:

1. **Data transformation**: PIO can byte-swap, pad, or preprocess data before feeding to SHA-256
2. **Multi-stream**: Multiple PIO SMs can feed different data sources to SHA-256 sequentially
3. **Custom protocols**: PIO reads from external pins (SPI, I2S, PIO-driven sensors) and feeds directly to SHA-256 without CPU or DMA
4. **Resource sharing**: When DMA channels are all in use, a PIO SM can act as a lightweight DMA controller

#### PIO Data Preprocessing Example

For SHA-256, input data must be big-endian. If data arrives little-endian (common on ARM), PIO can swap bytes on-the-fly:

```asm
.program sha256_pump_bswap
    pull block          ; Get little-endian word
    mov x, osr          ; Save to X register
    ; Byte swap via bit operations
    out null, 8         ; Discard byte 0
    out null, 8         ; Discard byte 1
    out null, 8         ; Discard byte 2
    in osr, 8           ; Re-read as big-endian
    out pins, 32        ; Output swapped word to SHA-256
.wrap
```

#### Known Limitations of PIO Path

- PIO has no ALU for bitwise operations — byte swapping requires creative instruction sequences
- PIO state machine clock divider limits throughput (max ~133 MHz / 1 = 133M words/sec)
- PIO program memory is limited (32 instructions max)
- Not currently implemented in the production firmware — the DMA path is sufficient for typical workloads

## C Bridge API

```c
const char* sage_sha256_hash(const uint8_t* data, uint32_t len);     // Word-at-a-time
const char* sage_sha256_hash_dma(const uint8_t* data, uint32_t len); // DMA if available
const char* sage_sha256_str(const char* s);                          // Null-terminated string
```

Returns pointer to static 65-byte buffer. Valid until next call.

## Known Limitations

- **Static output buffer**: Not thread-safe — single buffer shared across calls
- **No streaming API**: Must hash complete data in one call (no `update`/`finalize`)
- **PIO path not implemented**: DMA → SHA-256 is the production path
- **SHA-256 only**: RP2350 hardware only supports SHA-256 (no SHA-1, SHA-512, SHA-3)
